Pre-Authentication Traffic, Attacking RADIUS
Before access to the rest of the network, the client is allowed to communicate with the RADIUS server for authentication regarding the supported EAP type.
RADIUS servers have demonstrated a history of security vulnerabilities in handling malformed EAP traffic. The switch's only responsiblity is to forward EAP payload to the server encapsulated in a RADIUS Tyle/Length/Value (TLV) field.

EAPMD5 Fuzzing
If the RADIUS server stops returning RADIUS traffic back to the client, the attacker knows the RADIUS server has stopped responding. This can be dangerous causing DoS in a client environment, so maybe ask to fuzz the server in a non-production environment in-case of a crash.

EAP Shadow Attack
With a legitimate client, an attacker can "shadow" a device post-authentication by impersonating the MAC of a victim. 802.1X is a layer2 protocol, so the switch doesn't care about the IP address, only the MAC.
[PWN Plug] | Commercial plug-and-hack tool from Pwnie Express. Will bridgfe all traffic between two devices and watch for downstream HTTP activity. It will automatically copy a victim's MAC and IP and create an SSH tunnel for the attacker.

VLAN Manipulation
Windows do not natively support VLAN trunking features (such as IEEE 802.1Q). Windows will identify and drop tagged frames upon receipt instead of passing to the OS.
Virtual machine guests are unable to observe trunk packets and unable to participate in or exploit VLANs as well.

Use a native Linux OS system when working with VLANs as much as possible.
